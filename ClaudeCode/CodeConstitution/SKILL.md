# CodeConstitution — 编码宪法

AI 编码的最高约束框架。以 Karpathy 工程哲学为根基，定义「安全 → 可观测 → 可读 → 性能 → 极简」五层优先级栈，提供约束仲裁、审查锚点、遗留代码适配等元规则。适用于对代码质量有严格要求的团队和项目。

## When to use

在以下场景必须激活本 Skill：
- 生成任何生产环境代码
- 修改/重构现有代码
- 编写测试用例
- 架构设计/技术方案输出
- 生成 CI/CD 配置文件、Dockerfile、脚本

## Triggers

- 用户提及「宪法」「constitution」「编码规范」「约束」相关关键词
- 任务涉及生产环境代码生成或重构
- 任务要求"严格遵循规范"或"高质量代码"

## Companion files

本 Skill 包含以下文件：

- `SKILL.md` — 主文件，含完整编码规范
- `companion/checklist.md` — 代码生成后的自我合规检查清单

## The Rules

### P0 级（每次代码生成必须遵守）

#### P0-1：伪代码先行

强制 AI 先输出带注释的伪代码/流程图，经确认后再转译为真实代码。避免直接生成代码导致的逻辑黑盒、难以调试。

Karpathy：简单是默认值，但安全是底线。

#### P0-2：显式状态机

任何涉及状态流转的逻辑必须显式定义状态机（状态、事件、转换条件），禁止隐式状态变量。防止异步流程、订单流转、UI 交互中的状态竞态与死锁。

#### P0-3：边界条件防御

在生成主逻辑后，强制 AI 列出至少 5 个边界条件并逐一编写防御性代码。解决 AI"只考虑 Happy Path"的通病。

#### P0-4：结构化日志埋点

为关键逻辑分支添加结构化日志埋点，并解释每个分支的触发条件。避免生成"能跑但无法排查问题"的代码。

#### P0-5：强制 YAGNI 审查

在生成任何接口、基类、工厂模式或配置项之前，必须确认当前是否有 ≥2 个已确认的调用方/使用场景。否则使用最朴素的内联实现并标注 `# YAGNI: 待第二个用例出现时再抽象`。

#### P0-6：测试即规格

每个函数/方法的单元测试必须在同一代码块或紧邻位置输出，测试用例必须直接映射到伪代码中的边界条件和状态转换。禁止先写完整业务代码再补测试。

#### P0-7：依赖最小化声明

引入任何第三方库时必须说明"为什么不能用现有依赖或 10 行内联代码替代"。生成代码后验证未引入意外传递依赖。

#### P0-8：错误处理显式契约

所有 `try/except` 块必须满足：① 捕获具体异常类型；② 记录结构化日志（含上下文变量）；③ 明确声明该异常是"重试"、"降级"还是"终止"。禁止裸 `except:`。

#### P0-9：性能意图标注

选择非显而易见的算法/数据结构时，必须在注释中标注预期瓶颈和验证方式。禁止以"性能更好"为由引入复杂度，除非有量化依据。

#### P0-10：变更影响面声明

修改现有代码时，必须先输出影响分析：列出所有调用方、共享状态、外部依赖，并说明向后兼容性。

#### P0-11：数据不可变性优先

状态对象使用 `@dataclass(frozen=True)` 或 `NamedTuple`，集合使用 `tuple`/`frozenset`。若因性能需要可变对象，必须在注释中说明原因。

#### P0-12：资源生命周期显式化

文件句柄、网络连接、锁、数据库游标等必须使用 `with` 语句获取和释放。禁止依赖 `__del__` 或 GC 隐式释放。

#### P0-13：函数纯度标记

每个函数文档字符串必须包含 `Pure:` 或 `Impure:` 标记。Pure 标记的函数相同输入永远相同输出，无副作用。

#### P0-14：自我合规检查清单

在输出最终代码之前，必须输出一个检查清单，逐条确认本对话中所有已定义的约束，每条须标注满足的证据或解释不适用原因。

#### P0-15：硬编码零容忍

禁止字面量数字（除 0, 1, -1 及数组索引）、字符串、正则表达式模式。所有字面量必须定义为命名常量，并注释说明来源。

#### P0-16：统一编码风格

强制使用项目统一格式化规则。单行 Python 88 字符/其他 120 字符硬阈值。禁止混合缩进。分支/循环体即使一行也必须加代码块。

#### P0-17：命名强规范

变量/函数/类命名见名知意。禁止单字母（除循环 i/j）、拼音、模糊命名。函数用动词开头。布尔值用 `is_`/`has_` 前缀。常量全大写下划线。

#### P0-18：注释规范

代码做什么不用注释，为什么这么做必须注释。注释用词必须精确无歧义，同一概念全文使用统一术语。业务规则、历史兼容、workaround 必须加背景注释。弃用代码直接删除，禁止注释封存。

#### P0-19：日志分级规范

严格区分 TRACE/DEBUG/INFO/WARN/ERROR/FATAL。所有日志携带统一上下文。ERROR 仅用于真正异常，业务正常失败用 WARN。错误日志必须含异常栈和现场变量。

#### P0-20：环境状态快照

所有入口点（脚本、服务启动、任务函数）的前 10 行必须输出环境快照日志：运行目录、命令行参数、环境变量（脱敏）、Python 版本、操作系统。

#### P0-21：敏感数据零明文

密码、密钥、Token、手机号、身份证禁止硬编码、禁止明文打印日志、禁止明文传输。配置类敏感信息从环境变量/加密配置中心读取。

#### P0-22：输入全量校验

所有外部输入必须逐字段校验类型、范围、格式、长度。校验失败抛出明确业务异常。

#### P0-23：函数内分支出口收敛

单函数显式出口不超过 8 个。分散出口统一收拢。

#### P0-24：集合判空统一写法

统一使用语义化写法（`if not list_data`），禁止 `len()==0` 反复判断。先判空再取值。

#### P0-25：最小权限原则

代码内权限判定采用最小必需权限。所有入口强制校验用户身份和权限范围。禁止默认放行。

#### P0-26：会话与凭证有效期

所有 Token、会话必须有明确过期时间，支持主动吊销。禁止永不过期的凭证。过期后必须重新认证。

#### P0-28：加密算法强标

禁用弱算法（MD5、DES）。口令存储必须加盐慢哈希（bcrypt/argon2）。密钥定期轮换。签名验签全链路强制。

#### P0-29：文件/命令执行防注入

禁止字符串拼接系统命令，使用参数列表传参。文件权限设为最小可用。子进程路径和允许执行的命令做白名单限制。

#### P0-30：环境快照启动

入口处打印脱敏环境变量、关键配置、版本号。所有入口点（脚本、服务启动、任务函数）的前 10 行必须输出环境快照日志。

#### P0-31：死循环兜底

所有 `while` 循环、无限轮询必须设置最大执行次数/超时常量兜底。触发兜底时打印告警日志。

#### P0-32：卫语句优先

所有函数先校验非法入参、异常状态、边界条件并直接返回/抛出，再走主流程。禁止大包裹式 if 主逻辑。

### P1 级（涉及相关场景时激活）

#### P1-1：表达式复杂度管控

条件判断嵌套深度不超过 4 层。禁止超长复合条件，多条件 `and/or` 必须拆分。三元表达式仅允许简单值赋值。

#### P1-2：语法糖克制使用

不滥用列表推导式嵌套超 2 层、海象运算符、匿名函数多层嵌套。复杂逻辑优先命名函数。

#### P1-3：分支逻辑扁平化

优先使用卫语句（Guard Clause）：先校验非法入参、异常状态、边界条件并直接返回，再走主流程。

#### P1-4：类型强校验

核心代码启用静态类型检查。函数入参、返回值必须显式标注类型注解。字典优先用不可变数据类替代裸字典。

#### P1-5：空值显式处理

所有可能为空的变量标记可空类型。遵循"先判空、后使用"。禁止链式调用不判空。

#### P1-6：重复代码禁止（DRY）

连续 3 行及以上完全一致逻辑必须抽为独立函数。禁止复制粘贴后仅改变量名的"伪复用"。

#### P1-7：运算符优先级显式化

多运算符混合表达式必须用括号明确优先级。位运算、三目运算混合场景强制加括号。

#### P1-8：路径与文件系统跨平台兼容

禁止硬编码路径分隔符，使用 `pathlib`/`os.path`。脚本启动路径不依赖当前工作目录。

#### P1-9：异步/协程显式契约

严格区分同步/异步函数。`async def` 内部必须全部使用异步 IO。禁止同步代码中裸调用异步函数。

#### P1-10：锁与并发原语显式管控

所有锁获取/释放路径全路径可见，必须成对，优先使用上下文管理器。多锁场景固定加锁顺序。分布式锁必须设超时时间。

#### P1-11：全局可变状态严格管控

非必要禁止全局可变变量。全局状态修改点全部集中，禁止跨文件零散修改。全局状态变更必须打日志。

#### P1-12：递归使用强限制

业务逻辑优先迭代实现。递归必须显式设置最大深度阈值。深度超栈限制时改写为栈模拟迭代。

#### P1-13：单一职责严格落地

一个函数只做一件事：不混合 IO + 计算 + 状态修改。函数名出现 `and`/`or` 视为违规。

#### P1-14：工具函数纯函数强制

工具函数标记 Pure，禁止读写全局状态、IO、配置、日志。入参全部使用不可变类型。

#### P1-15：删除优于注释

禁止通过注释大段代码块保留"未来可能用到的代码"。必须直接删除，依赖 Git 恢复。唯一例外：`# EXAMPLE` 块。

#### P1-16：显式错误类型契约

函数失败必须在返回类型或异常声明中显式体现。禁止泛用 `Exception`。外部调用必须将底层异常映射为领域错误。

#### P1-17：深拷贝与浅拷贝的显式选择

在需要复制复杂结构体时，必须显式区分深拷贝/浅拷贝，并在注释中注明对内存/性能的影响。

#### P1-18：金额与高精度计算强制拒绝浮点数

涉及货币、金融资产的计算绝对禁止使用 `float`。必须使用 `Decimal` 或最小单位整数。

#### P1-19：时间字面量时区强制化

禁止创建"无时区信息"的时间对象。所有时间操作显式指定时区，优先 UTC。

#### P1-20：拒绝黑魔法与动态执行

严禁使用 `eval()`、`exec()`、`subprocess(shell=True)`。动态反射的 key 必须经白名单校验。

#### P1-21：上游重试风暴防御

拒绝非法请求时返回明确客户端错误码（400/429）。支持重试时必须建议退避时间。

#### P1-22：零占位符完整交付

禁止使用 `# ... 其余逻辑保持不变` 或仅输出修改片段。必须输出 100% 完整、可直接运行的代码。

#### P1-23：意图对齐与不确定性拦截

当需求模糊或与约束冲突时，AI 禁止盲猜。必须向人类抛出结构化确认问题，给出 A/B 方案。

#### P1-24：幂等性与唯一标识符契约

状态变更操作必须声明幂等策略。通过唯一标识符做冲突/重复检测防御。

#### P1-25：多存储/多网络操作的事务与最终一致性防御

跨存储操作必须画出故障隔离边界，编写补偿或降级逻辑。

#### P1-26：大上下文长文件滑动窗口记忆保护

长代码文件采用分块锚定输出：IMPORTS → TYPES → DOMAIN LOGIC → DEFENSIVE → TESTS。后半部分必须引用前半部分定义的常量。

#### P1-27：运行期状态自愈与死锁打破

长生命周期进程必须包含保底时间轮询或 Watchdog。状态机长期停留在中间状态时主动打破死锁。

#### P1-28：设计决策记录强制输出

在输出最终代码前输出设计决策记录，包含决策点、备选方案、选择理由（引用优先级栈）、代价/风险。

#### P1-29：规则按需剪裁与声明机制

根据任务属性自动执行"规则剪裁"：纯本地脚本聚焦边界防御和错误处理；微服务全量激活分布式约束。

#### P1-30：技术债引入的显式量化

违反低优先级约束时，在文件顶部生成技术债评级块：RISK + REMEDIATION + TRIGGER。

### P2 级（审查阶段检查）

#### P2-1：缓存策略显式契约

所有缓存显式声明：缓存 Key、有效期、淘汰策略、穿透/击穿/雪崩防御。

#### P2-2：重试机制标准化

网络/DB/外部调用重试必须定义最大次数、退避算法、可重试异常白名单。非幂等接口禁止自动重试。

#### P2-3：优雅启停与信号处理

常驻进程必须处理 SIGINT/SIGTERM，实现优雅退出：停止接收新请求 → 等待存量完成 → 释放资源。

#### P2-4：限流与熔断机制

对外接口必须内置限流逻辑。下游连续故障时触发熔断。阈值全部外置为配置。

#### P2-5：配置与代码彻底解耦

业务开关、超时、阈值、地址、路由等全部外置为配置。配置读取统一封装入口。

#### P2-6：环境区分隔离

禁止写死环境标识。测试环境专属逻辑必须加环境判断开关。生产环境自动关闭调试接口/日志。

#### P2-7：函数/接口入参出参契约化

公开函数必须明确定义入参类型、取值范围、返回值、异常种类。参数建议 ≤ 6 个。

#### P2-8：单向依赖原则

模块依赖方向严格单向。底层模块不允许反向依赖上层业务模块。

#### P2-9：测试分层规范

严格区分单元测试/集成测试/端到端测试。单元测试全量 Mock 外部依赖。测试目录与源码结构对齐。

#### P2-10：异常分支全覆盖测试

每种自定义异常和捕获的具体异常必须编写独立测试用例模拟触发。

#### P2-11：数据隐私合规掩码（PII）

日志中含 PII 必须使用统一脱敏函数掩码。手机号保留前3+***+后4，邮箱保留首字符+*@***。

#### P2-12：单一职责严格落地

一个函数/类/模块只做一件事。模块按业务领域划分，禁止"大杂烩工具模块"。

#### P2-13：健康检查与优雅启停

服务必须提供 `/health` 接口，包含依赖组件状态。进程停止时先停止接收新请求→处理存量→清理资源→退出。

#### P2-14：配置与代码彻底解耦

业务开关、超时、阈值、地址、路由等全部外置为配置，按环境分层管理。配置读取有统一封装入口，不散落在业务代码中。

#### P2-15：版本号内嵌

所有可独立部署的程序内置版本常量，启动时输出版本信息。依赖的第三方组件做最低版本校验，不满足直接终止。

#### P2-16：数据 Schema 兼容

存储结构变更只能新增字段并带默认值，禁止直接删除或重命名旧字段。代码必须同时兼容新旧两种数据格式。

#### P2-17：连接池显式管控

数据库、Redis、RPC 等连接统一使用连接池，参数（最大连接数、空闲超时、获取超时）配置化。连接耗尽时抛出明确业务异常。

#### P2-18：批量与分片强制

批量操作必须按常量分片执行（如每次 1000 条），禁止一次性加载全量数据到内存。失败子集单独重试，不整批重试。

#### P2-19：分布式链路透传

跨进程调用（HTTP、RPC、MQ）必须透传 TraceID 等全链路追踪上下文。禁止在网络调用边界丢失或截断上下文。

#### P2-20：慢操作阈值告警

数据库查询、外部 API 调用超过常量阈值时自动输出慢日志，记录完整调用栈和入参。

#### P2-21：告警分级降噪

异常按严重度分级（紧急/普通/仅日志）。短时重复异常合并触发，内容直指影响范围和处理建议。

#### P2-22：内存泄漏防范

长生命周期对象（单例、全局缓存、事件监听）必须提供清理/注销机制，并绑定到优雅退出路径。

#### P2-23：异步任务状态持久化

后台任务、异步队列执行状态（待执行/执行中/成功/失败）必须落地持久化。服务重启后能自动恢复未完成任务。

#### P2-24：通用高危红线

- 硬编码敏感信息彻底禁止
- 线上环境禁止开启调试模式
- 禁止直接拼接 SQL/命令语句
- 所有接口设报文大小上限
- 外部输入全量转义输出
- 后台服务禁绑定公网弱监听
- 禁止在循环内创建数据库/网络连接

### 约束优先级仲裁

当低优先级约束与高优先级冲突时：

```
安全/正确性 > 可观测性 > 可读性 > 性能 > 极简性
```

违反时必须在注释中声明豁免理由（`# PERF-OVERRIDE: 可变对象避免O(n)拷贝，见基准测试`）。

### 合规层级分级（渐进式激活）

本 Skill 的 P0/P1/P2 分级并非固定不变。根据任务类型自动激活：

| 任务类型 | 激活层级 |
|---|---|
| 纯本地脚本/工具 | P0 全量 + P1（部分） |
| 微服务/常驻进程 | P0 + P1 + P2 全量 |
| 修改遗留代码 | P0 全量 + 遗留代码适配协议 |

### 人类审查锚点

为关键决策点生成标准审查锚点注释：

```python
# REVIEW:C1-YAGNI: 未抽象Parser接口，当前仅CSV一种格式
# REVIEW:C4-ERR: FileNotFoundError→降级返回空列表，非终止
```

### 遗留代码适配协议

修改现有文件前先评估合规度。< 50% 时采用隔离+渐进策略：新增代码严守约束，不强制重构旧代码。
